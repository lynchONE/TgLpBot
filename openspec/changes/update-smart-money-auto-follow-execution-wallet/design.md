## Context
自动跟单的“目标钱包”是被监听的钱包，不是资金执行钱包。当前服务在构造跟单 `StrategyTask` 时调用默认钱包，导致多钱包用户无法知道或控制实际执行资金来源。

## Goals / Non-Goals
- Goals: 配置级保存执行钱包；开仓任务使用配置钱包；双端表单明确展示。
- Goals: 旧配置无执行钱包时保持可运行，并在读取接口中解析出当前默认执行钱包用于展示。
- Non-Goals: 不改动手动开仓、撤仓、核心流动性执行逻辑。
- Non-Goals: 不引入多执行钱包分摊或轮询策略。

## Decisions
- Decision: 使用 `execution_wallet_id` 和 `execution_wallet_address` 作为自动跟单专属字段。
  - 理由：避免和 `target_wallet_address` 混淆，也避免误复用 `StrategyTask.wallet_id` 的语义。
- Decision: 保存配置时校验执行钱包必须属于当前用户。
  - 理由：执行钱包涉及真实资金，不能用静默默认值掩盖错误。
- Decision: 历史配置读取与执行时用默认钱包兼容，但新保存请求如果显式传入非法钱包必须失败。
  - 理由：旧数据需要平滑运行，新配置必须明确可控。

## Risks / Trade-offs
- 风险：默认钱包变更会影响未设置执行钱包的历史配置。
  - Mitigation: 前端展示解析后的执行钱包；用户编辑保存后会固化为明确执行钱包。

## Migration Plan
1. 为配置、作业、映射表添加执行钱包字段。
2. 保存配置时写入执行钱包，读取列表时补齐未设置配置的解析钱包。
3. 自动跟单创建任务时优先使用配置执行钱包。
4. 前端加载钱包列表并提供选择器。
