## Context
当前 CLMM 任务越界逻辑由三组因素共同决定：价格方向（上破/下破）、`rebalance_enabled`、`stop_loss_enabled`。这让用户很难直接预测“超过区间后到底会发生什么”，也与开仓页的简化心智冲突。

用户的新目标是把越界执行模式统一成两类：
- 自动模式：开启再平衡，任意方向越界都在缓冲时间后自动再平衡。
- 手动模式：关闭再平衡，任意方向越界都在缓冲时间后自动撤仓并终止任务。

另外，单边池在技术上天然是“初始就处于区间外”，不能直接复用双边池的越界处理，否则任务刚创建就会触发自动动作。

## Goals / Non-Goals
- Goals:
  - 统一 CLMM 任务的越界执行语义，去掉“上破/下破不一致”和“只提醒不处理”的分叉。
  - 保持现有 `reopen_delay_seconds` 作为统一缓冲时间。
  - 让单边池在首次真正进入区间前保持静默，避免刚开仓就被系统自动处理。
  - 让 WebApp 与 MiniApp 的开仓页文案和实际后端行为一致。
- Non-Goals:
  - 本次不重构任务暂停 API 与任务停止 API 的整体交互。
  - 本次不删除数据库中的 `stop_loss_enabled` / `stop_loss_delay_seconds` 字段。
  - 本次不扩展到 AutoLP / SmartLP 的非 CLMM 路径。

## Decisions
- Decision: `rebalance_enabled` 成为 CLMM 越界策略的唯一执行开关
  - `true`：缓冲后再平衡。
  - `false`：缓冲后撤仓并终止任务。
  - `stop_loss_enabled` 仅保留兼容字段，不再作为 CLMM 越界处理分支条件。
  - Why: 这与用户心智最接近，也能消除当前“上破/下破行为不同”的隐式规则。

- Decision: 单边池采用“首次进入区间后再激活越界监控”的状态机
  - 为任务新增持久化状态，建议使用时间戳字段，例如 `range_activated_at`。
  - 若任务创建/重开时价格已在区间内，则立即写入激活时间。
  - 若任务创建/重开时价格仍在区间外，则不记录激活时间；只有价格首次进入区间时才写入。
  - 在 `range_activated_at` 为空时，单边池不会启动 `out_of_range_since` 倒计时，也不会触发自动再平衡或撤仓。
  - Why: 该状态需要跨轮询、跨重启保持，不能只靠内存推导。

- Decision: 手动模式的自动撤仓沿用现有“退出到 USDT 并终止任务”链路
  - 不新造一套“手动模式越界退出”执行框架。
  - 复用现有退出重试与最终 `finishStopAfterExit` 终态逻辑，只调整触发条件与文案原因。
  - Why: 现有退出链路已经具备重试、收尾和终止任务能力，复用风险更低。

- Decision: 双端开仓页改成单一越界模式说明
  - 不再把“止损”作为 CLMM 开仓页的主交互开关。
  - 改为展示“开启再平衡”或“关闭再平衡（超区间后缓冲撤仓终止）”。
  - 单边池附加提示：“首次进入区间前不会触发自动再平衡或撤仓”。
  - Why: UI 必须与后端真实行为一致，否则继续制造误解。

## Risks / Trade-offs
- 风险: 旧任务原本依赖 `stop_loss_enabled=true` 做“下破止损”的用户，升级后会变成“关闭再平衡才撤仓终止”。
  - Mitigation: 在 UI 与变更说明中明确新的统一规则；旧字段保留但不再驱动 CLMM 越界执行。

- 风险: 单边池新增激活状态后，需要在新开仓、再平衡重开、手动恢复等路径保持一致。
  - Mitigation: 将激活状态初始化收敛到任务创建和重开成功后的统一更新点，并补足测试。

- 风险: 仅调整开仓页而不调整其他说明文档，会导致 README / 管理页仍保留旧语义。
  - Mitigation: 至少同步更新关键用户文案；若本次不动管理页，需在提案实现说明中标记遗留点。

## Migration Plan
1. 新增任务激活状态字段，并在模型层与任务创建/重开成功路径中初始化。
2. 重写策略轮询中的越界判定分支，移除基于方向和止损开关的执行差异。
3. 调整 WebApp / MiniApp 开仓页文案与开关，统一心智。
4. 通过单元测试与双端构建验证行为一致性。
