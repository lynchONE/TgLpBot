## Context
- 当前开仓链路以 `AmountUSDT` 作为用户输入的目标仓位金额，但实际入场币可能是 `USDC`、`WBNB` 等任意 entry token。
- 现有实现会先读取钱包 `USDT` 余额并对 `AmountUSDT` 做截断，再去判断是否可以直接使用已有 entry token，顺序错误会让“已有入场币余额”失效。
- 现有热门池子数据源已经保存了 `current_pool_value`、`current_liquidity`、`active_liquidity_usd` 等字段，但开仓接口没有统一复用这些字段做分级风控。
- 本次还要求将低流动性风险前置到同一开仓页面中，不能只在后端报错后让用户自己猜原因。
- 旧的 `ZapSafety` 配置目前存在三个问题：
  - 配置默认值写死在代码里，数据库未配置时不会读取部署环境中的阈值。
  - 价格偏差检查主要挂在“存在 OKX swap quote”的路径上，已有入场币直接开仓时不会执行。
  - 最小流动性 USD 检查只覆盖部分 V3 路径，V4 和统一开仓页并没有真正复用。

## Goals / Non-Goals
- Goals:
  - 修正入场金额解析，避免仅因 `USDT` 余额较低而错误缩小实际开仓金额。
  - 为开仓接口提供统一的流动性分级判断与结构化响应。
  - 将历史上的价格偏差与最小流动性阈值合并到统一开仓前校验中。
  - 让 MiniApp / WebApp 能在原开仓弹窗中完成“看到提示 -> 修改金额 -> 知悉风险 -> 继续开仓”。
  - 在热门池子接口层彻底过滤超低流动性池子。
- Non-Goals:
  - 不改搜索池子结果的展示策略；手动搜索仍可返回池子，但开仓时仍要走同样的风控。
  - 不在本次变更中重新设计整套滑点、黑名单或 AutoLP 规则。
  - 不在本次变更中重做整套入场路由或报价算法；仅修正“已有入场币优先、单次兑换后按实际到账开仓”的执行语义。

## Decisions
- Decision: 开仓金额按“实际可用入场币 best-effort”执行，不做补缺口兑换
  - 用户输入的 `amount` 是预算上限，不是必须凑满的精确成交额。
  - 当钱包里已有入场币时，优先使用现有入场币开仓，且不得因为 `USDT` 余额较低而先把目标金额错误截断。
  - 当钱包里没有可直接使用的入场币且允许入场兑换时，系统最多执行一次正常兑换。
  - 兑换完成后，实际到账多少入场币，就按多少开仓；若低于用户输入目标值，也不得再额外发起一次补差兑换。
  - Alternatives considered:
    - 维持“先按 `USDT` 余额截断，再判断 entry token 是否可用”：会继续产生明显错误的缩仓行为，排除。
    - 在已有入场币不足目标金额时，再追加一次 swap 去补足差额：与用户要求的“换完多少就拿多少开，不补缺口”相冲突，排除。

- Decision: 开仓流动性阈值使用统一的“流动性美元值”口径，并保留链上 raw liquidity=0 硬拦截
  - 开仓时先读取链上当前 raw liquidity；若为 `0`，直接拒绝。
  - 在美元阈值判断上，优先使用可获得的当前流动性美元值：
    - 首选 `active_liquidity_usd`
    - 不可用时回退到 `current_pool_value`
    - 若内部目录快照不存在，再按需回退到搜索链路可获得的第三方 `liquidity.usd`
  - 若无法得到任何可用的流动性美元值，则按“无法确认流动性风险”拒绝开仓，避免继续放行未知池子。
  - Alternatives considered:
    - 仅使用链上 raw liquidity 数值做 200/1000 阈值判断：raw liquidity 不是美元口径，无法直接满足用户的 `200U/1000U` 规则，排除。
    - 仅使用 `current_pool_value`：会把“总池价值高但当前活跃流动性很低”的情况误判为安全，不够稳妥。

- Decision: `POST /api/open_position` 对低流动性返回结构化错误/警告
  - 新增可选请求字段 `ack_liquidity_risk`。
  - 当流动性 `< 200` 时，接口返回明确错误码和 `liquidity_usd`，客户端只展示禁止开仓提示。
  - 当流动性 `>= 200 && < 1000` 且请求金额 `> 200` 或未确认风险时，接口返回明确错误码、`liquidity_usd`、`max_open_amount=200`、`risk_ack_required=true`。
  - 客户端在同一开仓弹窗中保留当前表单，允许用户直接修改金额到 `<= 200`，勾选“已知低流动性风险”后再次提交。

- Decision: 历史 `ZapSafety` 阈值升级为统一开仓前校验的配置源
  - `zap_price_deviation_max_percent` 继续表示“池价 vs OKX 价”的最大允许偏差，但不再只依赖 OKX swap quote；当不存在 swap quote 时，改为用 OKX 市价推导 token 比价进行比较。
  - `zap_min_pool_liquidity_usd` 继续表示“硬性最小流动性门槛”；统一校验时实际使用 `max(zap_min_pool_liquidity_usd, 200)` 作为直接禁止开仓阈值。
  - 当数据库 `system_configs` 中未配置这两个值时，后端必须回退读取环境变量阈值，再回退到代码默认值。
  - Alternatives considered:
    - 保留旧 `ZapSafety` 只在 swap 路径生效：会继续出现“线上明明配置了，但直接持币开仓根本没检查”的问题，排除。
    - 让新的 `200/1000` 规则完全覆盖历史阈值：会丢失管理员已经在线上配置的门槛意图，排除。

- Decision: 热门池子接口在服务端过滤 `< 100U` 的池子
  - `GET /api/pools` 在构造响应前过滤流动性美元值 `< 100` 的项。
  - 该规则对默认热门池子、带 `token_address` 筛选、带 `include_pools` 的响应一视同仁。
  - 过滤发生在服务端，前端不再承担兜底隐藏逻辑。

## Risks / Trade-offs
- 对目录快照缺失的池子引入按需第三方流动性查询，会增加部分开仓请求的延迟。
  - Mitigation: 先查内部 `pools` 表，仅在缺失时才回退第三方数据源，并保留合理超时。
- 低流动性二级提示会让开仓交互多一步确认。
  - Mitigation: 仅在 `200 <= liquidity < 1000` 时触发，且直接在原弹窗中完成，不增加额外页面跳转。
- 热门池子过滤后，榜单数量可能少于旧版本。
  - Mitigation: 这是预期行为，优先保证安全性与数据质量。

## Migration Plan
1. 先扩展开仓接口的低流动性返回结构，并保持旧成功响应兼容。
2. 调整入场金额解析顺序与“单次兑换后按实际到账开仓”的执行逻辑。
3. 在 MiniApp / WebApp 开仓弹窗中接入结构化风险提示、金额限制与风险确认。
4. 在热门池子接口统一增加低流动性过滤。
5. 回归验证后，再视情况补充搜索池子链路的复用方法。
