## Context
- 聪明钱资产页当前把“总览汇总”“钱包列表”“钱包详情”“排行榜”放在一套接口模型里，但其中最慢的链路仍然是钱包实时资产计算，尤其是钱包余额字段会在多个页面和接口里被重复消费。
- 当前钱包列表已经具备服务端分页，但仍需要明确：所有聪明钱包余额相关入口都必须复用同一份按钱包缓存的实时资产结果，而不是各自直接查链。
- 当前排行榜展示的是昨天快照结果，天然适合做按日缓存；而钱包详情日历虽然已经能标识转账日，但盈亏值仍直接取资产差额，存在误导。

## Goals / Non-Goals
- Goals:
  - 明显降低聪明钱资产页的首屏与轮询耗时。
  - 让钱包总览、单钱包详情、余额 enrich 等入口统一复用 5 分钟钱包缓存。
  - 让排行榜复用按日缓存，而不是每次请求都重新组装。
  - 让转账日不再把资金划转误记为盈利或亏损，同时保留转账标识。
- Non-Goals:
  - 本次不重做整套聪明钱监控链路，也不引入新的外部存储。
  - 本次不改动普通用户“我的资产”页面的数据结构。
  - 本次不尝试识别普通转账的精确净额，只先保证“不把它算进盈亏”。

## Decisions
- Decision: 排行榜缓存按“快照日 + 指标”存储
  - 排行榜结果基于昨日与前日快照，因此缓存键必须包含 `snapshot_day` 和 `metric`。
  - 每日聚合任务完成后立即刷新新快照日排行榜缓存。
  - 新快照日缓存写入后，删除前一快照日的排行榜缓存，避免长期堆积。

- Decision: 钱包实时资产使用按钱包的 5 分钟 Redis 缓存
  - 缓存键包含链和地址，确保同一钱包在不同余额入口读到同一份实时资产结果。
  - 钱包总览、单钱包详情、列表余额 enrich 都优先复用这份缓存；缓存 miss 或过期后再回源查链并回填 Redis。
  - 保留 `force_refresh` 作为显式绕过缓存的手段。
  - 每日快照任务为了记录真实快照时间点的数据，允许直接读取 live 状态而不复用这份短期缓存。

- Decision: 总览汇总与分页钱包列表解耦
  - 汇总信息、历史曲线、窗口统计仍由现有聚合表 / 统计逻辑提供。
  - 当前页钱包列表不再要求和汇总复用同一批全量实时查链结果。
  - 如后续仍需进一步优化，可继续把汇总改造成独立缓存，但本次先确保“当前页只查当前页钱包”。

- Decision: 转账日盈亏口径使用“零计入 + 明确标识”
  - 若某钱包某日检测到普通转入或转出，则该日排行榜盈亏值与钱包日历盈亏值不得使用资产差额。
  - 钱包日历优先使用 `sm_lp_daily_stats.estimated_realized_pnl_usd` 作为盈亏值；若当天存在转账，则该日盈亏按 0 处理并展示转账标识。
  - 排行榜 `pnl` / `yield_rate` 指标在转账日按 0 处理，`participation` 指标不受影响。

## Risks / Trade-offs
- 5 分钟缓存会带来短时陈旧数据。
  - Mitigation: 保留手动刷新与 `force_refresh` 绕过缓存能力。
- 服务端分页后，前端搜索不能再默认基于一次性加载的全量钱包列表。
  - Mitigation: 搜索词一并传到后端，由后端对钱包列表做过滤并返回分页结果。
- 将转账日盈亏直接置零会牺牲一部分真实收益信息。
  - Mitigation: 先保证不误记盈利 / 亏损；后续如需要，再增加“转账净额剔除”能力。

## Migration Plan
1. 扩展管理端聪明钱总览请求参数与响应结构，继续以服务端分页返回钱包列表。
2. 在后端增加排行榜按日缓存与钱包实时资产 5 分钟缓存。
3. 统一钱包总览、详情和余额 enrich 的缓存入口，补充测试。
4. 调整日历和排行榜盈亏口径，补充单测。
5. 更新 MiniApp 和 webapp 的资产管理页，接入新分页与跳转逻辑。

## Open Questions
- 钱包总览搜索是否需要同时影响顶部汇总卡片；当前默认不影响，只过滤列表。
- 排行榜缓存是否需要预热多个 `limit` 组合；当前默认缓存完整列表，请求时再裁剪。
