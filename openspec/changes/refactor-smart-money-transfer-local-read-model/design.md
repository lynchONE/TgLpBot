## Context
- 当前聪明钱钱包详情会在请求时调用资产服务的“当日转账检测”逻辑，按“今日 0 点到现在”把时间窗映射成区块范围，再反查 ERC20 `Transfer` 日志；这条路径在快链上会拉出很大的区块跨度，既容易撞上 RPC 限制，也让接口结果依赖当下 RPC 状态。
- Smart Money watcher 已经具备稳定的 HTTP 轮询区块能力，并且每轮最多只处理一个较小的区块窗口；从架构上看，普通转账事件更适合和 LP 事件一样，在 watcher 阶段增量解析、落库，再由资产模块按日聚合读取。
- 当前本地缺少聪明钱包转账明细层，只有 `sm_wallet_daily_snapshots` 上的日汇总字段，因此“今天”的日历和告警不得不在读取时反扫链补数据。

## Goals / Non-Goals
- Goals:
  - 让聪明钱包普通转账事件在 watcher 轮询过程中增量持久化。
  - 让聪明钱盈利日历、今日详情和排行榜只依赖本地聚合结果，不再在请求时扫描整日区块。
  - 保留现有转账标识、转入/转出次数与估算金额展示能力。
  - 延续“转账日不计入盈利/亏损”的业务口径，并确保口径建立在本地事件聚合之上。
- Non-Goals:
  - 本次不为历史日期做链上回补；功能上线前的旧日期沿用已有快照结果。
  - 本次不尝试做到精确的“转账净额剔除”，仍以“转账日归零 + 明确标识”为主。
  - 本次不新增聪明钱转账事件的独立前端页面或明细查询 UI。

## Decisions
- Decision: 引入聪明钱包转账明细表 `sm_wallet_transfer_events`
  - 每条记录表示一个钱包视角下的普通转账事件，至少包含 `wallet_address`、`chain_id`、`direction`、`asset_type`、`token_address`、`amount_raw`、`amount_decimal`、`amount_usd`、`tx_hash`、`block_number`、`log_index`、`tx_timestamp`。
  - 需要为“同一钱包 + 同一 tx + 同一 log 只写一次”建立唯一约束，保证 watcher 重试不会产生重复记录。
  - 事件表作为日聚合的 source of truth，`sm_wallet_daily_snapshots` 只保留按日汇总结果。

- Decision: 在 watcher 的增量轮询窗口里同步解析并持久化转账事件
  - ERC20 转账使用 watcher 当前处理的同一批区块窗口按钱包 topic 查询 `Transfer(address,address,uint256)` 日志，而不是等接口读取时再按整日扫链。
  - 原生币转账直接复用 watcher 已经拉取的区块交易列表，通过 `from`、`to` 与 `value` 识别入账/出账。
  - 为了延续现有盈利口径，需要排除 LP add/remove 同 tx 造成的内部 token 流转，避免把做 LP 的资金移动误标为普通转账。

- Decision: 资产管理改为只读取本地转账事件聚合
  - 历史日历继续以 `sm_wallet_daily_snapshots` 和 `sm_lp_daily_stats` 为主，但这些快照中的转账字段由本地转账事件表聚合得出。
  - “今天”的钱包详情不再调用按日扫链检测逻辑，而是读取 `sm_wallet_transfer_events` 当日聚合结果并和当前资产、当日 LP 统计合并。
  - 排行榜快照日盈亏同样只使用快照表中的本地聚合转账标识，不再依赖请求内临时补算。

- Decision: 历史回补不作为本次前置
  - 新读模型从部署后开始持续积累；未回补的旧日期继续使用既有日快照字段。
  - 如果后续需要提高历史完整性，可单独增加按日回补任务，但不阻塞本次移除请求时扫链。

## Risks / Trade-offs
- 新增明细表后，watcher 每轮会多一次钱包转账解析与写库，写入量和存储量都会增加。
  - Mitigation: 仅对已激活钱包解析；保持 watcher 小窗口轮询；通过唯一索引和按日聚合减少重复写入影响。
- 转账金额 USD 仍是按写入时的 best-effort 估值，不是严格历史成交价。
  - Mitigation: 当前业务主要需要“转账日标识”和粗略金额参考，这与现有 LP 事件金额估算口径保持一致。
- 新发现的钱包如果是在某轮扫描中途被加入监控，普通转账事件默认从其进入监控后的后续区块开始积累，不保证回补发现前的同窗历史转账。
  - Mitigation: 当前核心目标是消除请求时整日扫链；发现前历史回补可在后续独立能力中处理。

## Migration Plan
1. 新增聪明钱包转账明细模型、索引与数据库迁移。
2. 扩展 watcher，在增量区块轮询中持久化原生币与 ERC20 普通转账事件，并排除 LP 同 tx 内部流转。
3. 在资产服务中新增基于本地事件表的按钱包/按日聚合读取，替换钱包详情“today transfer detection”实时扫链路径。
4. 调整每日快照与排行榜读取逻辑，统一复用本地聚合结果。
5. 补充 watcher 和资产服务测试，验证去重、聚合与“转账日归零”口径。

## Open Questions
- 是否需要在转账明细表中额外持久化 `counterparty_address` 以便后续做流向分析；当前不是本次前置。
- 是否需要在部署后补一个一次性的“当天回补”脚本，填平上线当日 watcher 启动前的空窗；当前先不纳入本次范围。
