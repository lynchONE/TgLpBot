## Context
- 当前开仓与撤仓链路里存在多处固定等待，例如开仓后的余额等待、dust 二次读取等待、approve 后 allowance 固定重查，以及撤仓阶段统一按较长窗口等待 receipt 和余额同步。
- 当前撤仓失败后的再次尝试节奏也偏保守且不够分层，无法兼顾“BSC 正常时尽快补救”和“异常持续时避免空转重试”。
- 这些等待原本是为了兼容公共 RPC 滞后，但在 BSC 正常状态下往往会额外吞掉数百毫秒到数秒。
- 当前开仓预览依赖完整表单参数，导致池子检测不能在弹窗打开时启动；而后端每次预览/执行都会重新拉取部分池子上下文，重复 RPC 较多。
- 交易记录、交易历史和部分补账属于展示与追踪层面的非关键写入，不应阻塞已经成功的链上流程。

## Goals / Non-Goals
- Goals:
  - 把开仓和撤仓的关键路径缩短到“只覆盖链上成功判定和必要状态保护”。
  - 在打开开仓界面时立即返回池子基础检测结果，减少用户等待感。
  - 让预览与执行尽量复用已得到的池子元数据、tick、风控结果和 receipt，减少重复 RPC。
  - 针对 BSC 默认采用更激进的快路径轮询，但仍保留失败时的安全回退。
- Non-Goals:
  - 不试图缩短链本身的出块和确认时间。
  - 不在本次变更中重写 Zap 合约或大改入场/撤场链上结构。
  - 不把任务主状态更新完全异步化；任务状态仍需要同步落库以保证后续重试和界面状态正确。

## Decisions
- Decision: 新增 `open_position_prepare` 预检测阶段，专门负责“界面一打开就查”
  - `open_position_prepare` 只依赖 `chain`、`pool_address`、`pool_version` 和已确定的钱包上下文。
  - 当多钱包模式尚未选择钱包时，接口先返回与池子本身相关的检测结果，并显式标记仍需补齐钱包相关检查。
  - 该阶段返回基础池子信息、基础风控结果、推荐默认值以及可供后续预览/执行复用的上下文标识。
- Decision: `open_position_preview` 与 `open_position` 复用 prepare 结果
  - 新版客户端在用户继续填写金额、区间和滑点后，带上 prepare 上下文请求预览与执行。
  - 后端优先复用 prepare 阶段缓存的池子元数据和基础检查结果，只重新计算与金额、区间、滑点直接相关的部分。
  - 若 prepare 上下文缺失、过期或与当前参数不匹配，后端必须自动回退到完整计算，保证旧客户端兼容。
- Decision: 开仓与撤仓拆分“关键路径”和“异步补写路径”
  - 同步关键路径只保留：发交易、确认 receipt、必要的 allowance/余额保护、任务主状态更新。
  - 异步补写路径负责：交易记录、交易历史、非关键通知、dust/余额补账、额外 transaction 明细同步。
  - 异步补写失败时必须记录日志，并具备至少一次重试能力；接口返回不能因为这些失败而把链上成功误报为失败。
- Decision: 用快路径轮询替代固定等待
  - 开仓后余额读取、dust 读取、approve 后 allowance 校验、撤仓 receipt 读取和 token balance 同步，都先走短轮询快路径。
  - 仅当快路径窗口内未读到预期状态时，才退回现有较保守的等待策略。
  - 快路径参数按链区分，BSC 默认更激进，其他链可继续沿用保守默认值。
- Decision: 撤仓失败重试采用固定梯度退避序列
  - 撤仓失败后的任务重试间隔改为固定梯度：`500ms -> 1s -> 2s -> 3s -> 5s -> 10s -> 30s`。
  - 系统必须按失败次数向后推进重试档位，并在达到最后一档后继续使用 `30s` 作为稳定重试间隔，直到任务终止、人工干预或成功。
  - 对于可立即判定为无需重试的失败类型，系统仍可直接终止，不强制进入梯度退避。
- Decision: 尽量复用已获得的链上结果
  - 撤仓流程对同一批交易哈希只读取一次 receipt，并把结果复用于 gas 统计、USDT delta 统计和后续记录补写。
  - 开仓与撤仓过程中的 stable balance snapshot 在关键路径内采集，但其持久化动作进入异步补写。

## Risks / Trade-offs
- 异步补写会带来短暂的最终一致性窗口。
  - Mitigation: 任务主状态仍同步更新；交易历史允许短暂延迟，并对补写失败做重试和日志告警。
- prepare 上下文缓存会增加服务端状态与失效处理复杂度。
  - Mitigation: 使用短 TTL 和参数签名校验，失效后自动回退到完整计算。
- 更激进的 BSC 快路径可能在个别 RPC 节点抖动时更容易命中回退分支。
  - Mitigation: 保留原有保守路径作为 fallback，不以“快”为代价牺牲资金安全。
- 梯度退避会改变撤仓失败时的任务节奏，可能影响现有日志与监控判断。
  - Mitigation: 统一输出当前失败次数、当前档位和下一次重试间隔，便于排查。

## Migration Plan
1. 先在后端引入 `open_position_prepare` 与短时上下文复用能力。
2. 再把开仓/撤仓里的非关键记录写入改为异步补写，并接入重试。
3. 再把开仓固定等待、allowance 重试和撤仓同步等待改为快路径轮询加回退。
4. 接入撤仓失败梯度退避策略，并验证失败计数、档位推进和最终稳定在 `30s` 的行为。
5. 最后接入 WebApp 和 MiniApp，在弹窗打开时立即触发预检测，并回归验证旧客户端兼容链路。

## Open Questions
- 异步补写是否需要落 Redis/数据库队列做跨进程恢复，还是先采用进程内队列加失败重试。
- prepare 上下文是否需要显式 `prepare_id`，还是直接按用户、钱包和池子做服务端短时缓存命中。
