## Context
当前聪明钱钱包头像的来源是前端按钱包地址哈希后，从仓库内置头像集中选一个随机图标。这种方案不需要后端参与，但也意味着：

- 钱包不能拥有稳定、可识别的自定义头像。
- WebApp 和 MiniApp 无法共享一个由用户维护的头像资源。
- 一旦未来扩展到更多钱包管理场景，随机图标会继续成为识别噪音。

这次变更同时涉及数据模型、对象存储、HTTP 上传接口以及两个前端入口，属于典型的跨模块变更。

## Goals / Non-Goals
- Goals:
  - 支持为聪明钱钱包手动上传头像，并将头像绑定到钱包记录。
  - 上传文件统一存储到已部署的 MinIO `avatar` bucket。
  - WebApp 和 MiniApp 在同一钱包上展示同一张头像。
  - 未绑定头像的钱包继续兼容现有随机图标逻辑，不影响现有展示。
- Non-Goals:
  - 本次不引入头像裁剪、旋转、美化等图片编辑能力。
  - 本次不做头像版本历史、回滚或多图管理。
  - 本次不改造非聪明钱模块的钱包头像来源。

## Decisions
- Decision: 在 `monitored_wallets` 中新增头像字段
  - 为聪明钱钱包模型增加头像存储字段，用于记录该钱包当前绑定的头像资源。
  - 字段可以存储完整可访问地址，或存储对象键并由服务端拼接访问地址；实现时优先选择对前端改动更小、部署更直接的方式。

- Decision: 新增 MinIO 配置项并在服务端集中上传
  - 在配置文件中增加 MinIO Endpoint、Access Key、Secret Key、Bucket、是否 SSL、对外访问基地址等参数。
  - `avatar` bucket 作为默认头像存储桶，避免把 bucket 名写死在业务代码里。

- Decision: 上传接口与钱包绑定分两步或一步完成，但外部行为必须清晰
  - 服务端需要提供可供 WebApp / MiniApp 调用的头像上传入口，接收图片文件，完成校验与上传。
  - 上传成功后，钱包记录必须绑定该头像；实现时可以选择“上传时直接带钱包地址完成绑定”，也可以选择“先上传、再调用钱包更新接口绑定”，以代码最简且可维护为准。

- Decision: 展示层始终优先自定义头像，失败时回退本地图标
  - 钱包头像组件优先渲染钱包已绑定头像。
  - 若头像为空、加载失败或后端未返回，则回退到当前按地址散列的随机本地图标，确保现有界面兼容。

- Decision: 对上传文件做基础校验
  - 只接受常见图片格式，例如 PNG、JPG、WEBP。
  - 限制单文件大小，避免把超大原图直接推到对象存储。
  - 服务端以钱包地址、时间戳或随机后缀生成对象键，避免覆盖冲突。

## Alternatives Considered
- 方案 A：继续只用前端本地图标
  - 优点：零后端改动。
  - 缺点：无法满足手动上传和跨端共享头像的需求。
  - 结论：拒绝。

- 方案 B：把头像以 Base64 存在 MySQL
  - 优点：不用接对象存储。
  - 缺点：数据库膨胀快，传输与缓存都不合适，也违背用户已经准备好的 MinIO 部署。
  - 结论：拒绝。

- 方案 C：服务端上传到 MinIO，钱包记录持久化头像引用
  - 优点：符合现有部署条件，跨端共享简单，后续也方便做 CDN / 域名接入。
  - 缺点：需要新增对象存储依赖与配置。
  - 结论：采用。

## Risks / Trade-offs
- MinIO 访问地址配置错误会导致前端头像无法显示
  - Mitigation: 配置项显式区分服务端连接地址与对外访问基地址，并在接口层返回可直接展示的头像地址。

- 未做图片压缩时，用户上传超大图片会增加带宽和存储消耗
  - Mitigation: 首版先限制文件大小；如后续有需要，再补服务端压缩或裁剪。

- 钱包编辑接口当前只支持标签和启停状态，新增头像后需要保证旧调用兼容
  - Mitigation: 保持原字段兼容，仅增量开放头像相关字段或新增独立上传接口。

## Migration Plan
1. 为聪明钱钱包模型增加头像字段，并补齐 AutoMigrate / 历史列补偿逻辑。
2. 在配置结构中增加 MinIO 参数，初始化对象存储客户端。
3. 增加头像上传与钱包绑定接口，返回前端可直接使用的头像地址。
4. WebApp 接入头像上传与展示。
5. MiniApp 接入头像上传与展示。
6. 通过接口测试、Go 测试和前端构建验证行为。

## Open Questions
- 头像字段最终保存完整 URL 还是对象键，取决于现有配置加载和 URL 拼接成本；实现时按仓库现状选择更简洁方案。
- 是否需要覆盖上传旧头像时清理 MinIO 旧对象，本次可先不做强制删除，避免引入额外失败路径。
