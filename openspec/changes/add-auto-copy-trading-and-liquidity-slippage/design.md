## Context
当前补充流动性接口只接收任务 ID 与补仓金额，实际滑点来自任务已有配置。用户在行情波动或池子深度变化时，需要临时调整本次补仓滑点。

聪明钱模块已经具备钱包、池子、仓位、监控通知和特别关注能力，后端也有 `sm_lp_events` 与 `sm_lp_active_positions` 作为 LP 事件和当前仓位读模型。自动跟单会触发真实资金交易，必须以显式配置、可追踪任务和保守默认值实现。

## Goals / Non-Goals
- Goals:
  - 补充流动性支持本次滑点覆写，未填写时沿用当前仓位滑点。
  - 在聪明钱模块提供独立“自动跟单”页签。
  - 自动跟单支持目标钱包、金额模式、延时模式和撤仓跟单配置。
  - 每笔自动跟单任务可以追溯到触发的聪明钱事件与用户配置。
- Non-Goals:
  - 不做历史事件回放跟单。
  - 不把用户手动仓位或 AutoLP 仓位纳入撤仓跟单。
  - 不在首版支持复杂风控组合、随机延时、多级止盈止损或跨链跟单。

## Decisions
- Decision: 补流动性滑点使用请求级可选覆写。
  - `POST /api/task/add_liquidity` 增加可选 `slippage_tolerance`。
  - 为空时使用任务已有 `slippage_tolerance`；不为空时只影响本次补流动性执行，不自动修改任务长期滑点配置。
  - 后端必须在报价、模拟和链上交易构造中使用最终滑点值，避免前端展示值和实际执行值不一致。

- Decision: 自动跟单配置独立建模。
  - 新增用户级跟单配置表，按 `user_id + chain_id + target_wallet_address` 唯一约束。
  - 配置字段包含启用状态、目标钱包、金额模式、固定金额、比例、延时模式、延时秒数、是否跟随撤仓、游标事件 ID 或区块位置。
  - 金额模式枚举：`fixed`、`ratio`。
  - 延时模式枚举：`immediate`、`fixed_delay`。

- Decision: 自动跟单执行使用事件驱动的作业表。
  - 后端 worker 读取 `sm_lp_events` 中目标钱包的新 `add` / `remove` 事件，生成待执行作业。
  - 作业表记录配置 ID、事件 ID、动作类型、计划执行时间、状态、错误信息、创建的任务 ID。
  - 对同一配置和同一事件建立唯一约束，避免重复执行。

- Decision: 开仓跟单复用现有开仓/任务创建链路。
  - 跟单开仓使用被跟单事件的池子、协议、tickLower、tickUpper。
  - 固定金额模式按用户填写的 USDT 金额开仓。
  - 比例金额模式按 `被跟单事件开仓总 USD * ratio` 计算本次 USDT 金额；如果事件缺少可用开仓金额，作业必须失败并记录原因，不使用安全默认金额。
  - 新建任务需要标记来源为自动跟单，并持久化与目标事件的映射。

- Decision: 撤仓跟单只作用于自动跟单创建的对应仓位。
  - 配置关闭撤仓跟单时，remove 事件只记录为已跳过，不发起撤仓。
  - 配置开启撤仓跟单时，worker 按配置、目标钱包、池子、协议、tokenId 或 position ref 查找此前创建的跟单任务。
  - 找不到对应跟单任务时，作业失败并记录原因，不尝试撤出其他任务。

## Risks / Trade-offs
- 自动跟单会执行真实交易，风险高于提醒功能。
  - Mitigation: 默认关闭、必须显式启用；作业和任务保留触发来源；撤仓跟单单独开关。
- 比例金额依赖聪明钱事件中的 USD 估值。
  - Mitigation: 缺少估值时失败并提示，不用固定金额或 0 金额兜底。
- 延时跟单可能遇到池子状态变化。
  - Mitigation: 执行时复用现有开仓预检、滑点与链上模拟；失败时记录作业错误。

## Migration Plan
1. 新增自动跟单配置、作业、任务映射模型，并纳入 AutoMigrate。
2. 增加自动跟单配置 API 与 worker。
3. MiniApp 聪明钱模块新增“自动跟单”页签。
4. 补流动性 API 与 UI 增加可选滑点字段。
5. 增加针对滑点默认/覆写、金额模式、延时调度、撤仓跟单隔离的测试。

## Open Questions
- 跟单钱包是否首版只允许从已监控钱包中选择，还是允许输入任意地址后自动加入监控列表；本提案默认允许输入任意地址，但后端必须要求地址合法。
- 比例金额是否需要最小/最大单笔金额保护；用户当前需求未要求，首版先不加入隐式限制，后续可单独增加。
