## Context
- 当前资产管理模块已经覆盖用户资产、聪明钱资产、运行管理和系统管理，但 `miniapp` 与 `webapp` 的资产页都采用“整页并发请求 + 整页 loading”模式。
- 两端在资产模块中都会同时请求总览、历史与 LP 统计，且在轮询时反复触发 loading，导致图表和长列表重复重建。
- 用户已经确认允许 `1 分钟内` 的资产数据缓存，不要求秒级绝对实时，但手动刷新仍需要拿到最新结果。
- 本次范围要求保持现有图表渲染方案和头像方案不变，只优化请求、缓存和装载方式。

## Goals / Non-Goals
- Goals:
  - 降低 `miniapp` 与 `webapp` 资产管理模块的首屏等待时间和自动轮询抖动。
  - 为资产接口增加统一的 `60s` 短期缓存，减少重复计算与重复请求。
  - 保留已有数据时采用静默刷新，避免图表与列表闪烁。
  - 让管理员子页按需装载，避免阻塞普通用户资产页首屏。
- Non-Goals:
  - 不替换 `lightweight-charts`。
  - 不更换钱包头像资源和映射策略。
  - 不新增新的资产口径或图表样式。

## Decisions
- Decision: 资产接口采用 Redis 短期响应缓存
  - 用户资产接口与管理员聪明钱资产接口统一缓存最终响应 JSON。
  - 缓存 TTL 固定为 `60s`。
  - 缓存 key 必须包含用户身份或管理员身份以及会影响结果的请求参数。
  - 请求携带 `force_refresh=true` 时必须绕过缓存并回填最新结果。

- Decision: 前端采用“首屏加载 + 后台静默刷新”
  - 当页面尚无结果时，允许展示 skeleton 或空态 loading。
  - 当页面已存在最近一次成功结果时，自动轮询不得清空现有卡片、图表或列表，只能在后台刷新完成后替换数据。
  - 手动刷新允许展示按钮 loading，但页面主体仍保留最近一次成功结果。

- Decision: 资产页请求按优先级拆分
  - 用户资产页优先请求 `overview`，以尽快渲染核心指标卡片。
  - `history` 与 `lp_stats` 可并行在后台补齐，不再阻塞首屏主卡片展示。
  - 管理员聪明钱页继续区分 `overview`、`leaderboard`、`wallet detail`，但只在对应页签和选中钱包可见时发起请求。

- Decision: 双端入口改为按需装载
  - `miniapp` 与 `webapp` 的资产管理模块入口改为懒加载。
  - 管理员子区域只在激活对应页签后加载相关代码和数据。
  - 本次优先减少“非资产视图仍装载资产模块”的首屏成本，不做更激进的 chunk 拆分策略。

## Risks / Trade-offs
- 风险：缓存会带来最多 `60s` 的陈旧结果。
  - Mitigation: 手动刷新显式绕过缓存，自动轮询与缓存 TTL 对齐。
- 风险：前端保留旧数据后，用户可能误以为当前数据已完成刷新。
  - Mitigation: 保留按钮刷新态和最近更新时间，明确刷新中的状态。
- 风险：缓存 key 设计不完整可能导致跨用户串数据。
  - Mitigation: key 必须包含用户或管理员维度，以及窗口、榜单指标、钱包地址等请求参数。

## Migration Plan
1. 为资产相关接口增加 `force_refresh` 请求字段和 Redis 短期响应缓存。
2. 调整 `miniapp` 与 `webapp` API 层，支持带 TTL 的前端内存缓存与手动刷新绕过。
3. 重构资产页面刷新流程，将“整页 loading”改为“首屏 loading + 静默刷新”。
4. 将资产管理模块入口和管理员子区域切换为按需装载。
5. 运行双端构建和后端测试，确认页面功能与缓存语义一致。
