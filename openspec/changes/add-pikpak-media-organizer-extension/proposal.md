# Change: PikPak 媒体整理 Chrome 扩展

## Why
- 用户希望在 PikPak Web 中批量整理视频，当前去重、保留中文字幕版本、按年份归档、按主演女优归档都只能手工完成，成本高且误删风险高。
- 用户进一步要求“番号一致就算重复”，说明去重标准更贴近资源整理语义，而不是只看内容指纹。
- 用户还希望自动清理每个文件夹中小于 `100MB` 的小文件，用于删除样片、预览片段或低价值残留文件。
- PikPak 官方帮助中心说明 WebDAV 目前仅支持读取，不支持写入、复制和移动，因此无法用通用 WebDAV 客户端完成删除、建目录和移动整理，必须依赖已登录 Web 会话内的页面自动化能力。
- 用户明确要求“无法分类可以不操作”，这意味着插件需要把高置信度自动操作和低置信度人工复核严格分开。

## What Changes
- 新增一个独立的小型 Chrome 扩展项目 `chrome-pikpak-media-organizer/`，用于在用户已登录的 PikPak Web 页面上优先扫描当前页面已渲染的文件与目录元数据，并对可识别番号的视频生成整理计划。
- 扩展生成“整理计划”，覆盖以下动作：
  - 从文件名中提取并标准化番号
  - 联网查询番号对应的女优信息
  - 将“番号一致”的文件视为重复组并生成删除候选
  - 在番号重复组内优先保留带“中文字幕”标识的版本，并把 `番号-C`、`番号_C` 等常见后缀作为判断条件之一
  - 在仍需二次决策时优先保留体积更大的版本
  - 清理每个文件夹内小于 `100MB` 的小文件，不限制文件类型
  - 以“源目录”为处理单位，优先迁移其中识别出的目标长视频，并将同源目录中剩余文件纳入清理候选
  - 按“添加时间”的年份创建或复用目标目录
  - 使用女优中文名称作为二级目录名进行归档；若无法可靠识别女优中文名，则退回到年份目录
  - 在移动和删除完成后清理已经腾空的原始文件夹
- 扩展提供规则配置能力：
  - 中文字幕识别关键词
  - 番号提取规则
  - 在线元数据源顺序与请求策略
  - 女优中文名覆盖规则（支持 JSON 导入/导出）
  - 小文件阈值与清理策略
- 扩展提供预览与执行两阶段流程：
  - `Dry Run`：只生成计划，不执行删除和移动
  - `Apply`：用户确认后批量执行删除、建目录和移动
  - `Direct Execute`：用户确认后按当前“递归扫描”设置直接整理当前目录，不先停留在预览
  - 在扫描与执行过程中展示实时进度、运行日志，以及待删除文件/空目录数量和待确保目录名称
- 对无法可靠判断的文件默认不操作，并在预览中标记为“跳过”或“人工复核”。

## Scope Assumption
- V1 仅面向用户已经登录的 PikPak Web 页面；扩展不负责登录，不采集也不存储账号密码。
- V1 仅使用页面可获得的元数据进行判断，例如文件名、添加时间、大小、目录 ID、文件 ID、页面或接口返回的内容指纹/hash；不做视频内容 OCR、字幕流解析或封面识别。
- V1 中“保留中文字幕版本”仅基于文件名、标签或预设关键词判断，例如 `中字`、`中文字幕`、`CHS`、`C字幕`，以及 `番号-C`、`番号_C` 这类常见后缀。
- V1 中“重复”默认指在当前扫描范围内提取到相同标准化番号的文件组；若文件无法提取番号，则不参与自动去重。
- V1 中番号去重组的保留优先级默认为：`中文字幕命中 > 文件体积更大 > 添加时间更早 > itemId 字典序更小`。
- V1 中“小于 100MB 删除”默认对所有文件生效，并且不覆盖番号去重中已经选中的保留文件。
- V1 默认目录结构为 `年份/女优中文名/文件`；若能识别年份但无法可靠得到女优中文名，则默认移动到 `年份/文件`。
- V1 中女优识别优先通过在线番号元数据查询获得，用户本地覆盖规则仅用于修正中文名或补救查询失败场景。
- V1 会优先按“存在目标长视频的源目录”生成清理计划，并在执行阶段删除这些源目录中未被选中的剩余文件；仅在再次确认目录没有剩余子项时才执行目录删除。
- V1 不承诺跨账号、跨团队盘或跨外链整理，只处理当前登录用户在 PikPak Web 中可访问的个人文件视图。

## Risks / Tradeoffs
- PikPak Web 的页面结构和私有接口都存在变更风险，因此扩展在扫描阶段改为优先依赖当前页面已渲染的 DOM 列表，并在 DOM 不足时回退到页面内部请求。
- 当用户开启递归扫描且页面会话可用时，扩展需要优先走可递归遍历的页面会话结果；若会话不可用，只能回退到当前页面已渲染的 DOM 列表并明确提示结果不完整。
- 删除和移动属于高风险操作，V1 虽然支持按番号判重、按阈值删除小文件和清理空目录，但仍必须在执行前给出清晰预览，并允许用户逐条审阅。
- 不同来源的同番号文件可能并非完全等价，例如不同清晰度、不同字幕、不同片段版本，因此保留策略需要可解释且可配置。
- 在线女优元数据源可能限流、反爬、字段变更或偶发缺失，因此需要做本地缓存、失败回退和人工覆盖。
- “中文字幕”判断若只靠文件名与 `番号-C` 这类后缀会存在误判，因此该规则只能用于重复组内的保留优先级，不能作为内容真实性保证。
- 小于 `100MB` 的文件不一定都是样片，也可能是短视频正片或压缩版本；因此该规则必须显式展示命中原因，并默认不删除已选中的保留项。
- 空目录清理依赖执行后的实时校验；如果 PikPak 页面返回存在隐藏子项、延迟刷新或接口语义变化，扩展必须跳过目录删除而不是盲删。
- 不同元数据源返回的女优名称可能是日文、罗马字或中文，若无法稳定解析到中文名，则需要回退到本地中文映射或放弃女优分类。

## Open Questions
1. V1 默认采用 `年份/女优中文名/文件` 的目录结构，并在无法可靠得到女优中文名时退化为 `年份/文件`；如果你希望“拿不到中文名就完全不移动”，实现前需要把规则收紧。
2. V1 默认让“小于 100MB 删除”不覆盖番号组内已选中的保留文件；如果你希望它连保留文件也删，需要单独放开这个危险开关。
