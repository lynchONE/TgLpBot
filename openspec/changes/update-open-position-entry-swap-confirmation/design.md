## Context
- 当前开仓链路会先根据池子决定 entry token；像 `USDC/WBNB` 这类池子，系统通常会先做 `USDT -> USDC`，再把 `USDC` 用于后续开仓。
- 现有 `open_position` 请求如果没有显式传 `slippage_tolerance`，后端会直接使用 `cfg.SlippageTolerance`；这对稳定币到稳定币的 entry swap 过于粗糙。
- 当前页面没有 entry swap 预览阶段，用户看不到“这次会先换什么币、预计能收到多少、当前滑点是多少”，也无法在真实兑换前显式确认。
- 现有执行链路会在一次请求里直接进入“前置兑换 + 开仓”，且较早创建任务；如果只是想先看 entry swap 预估，不应该产生持久化副作用。

## Goals / Non-Goals
- Goals:
  - 统一开仓总流程，确保所有风险校验都发生在任何兑换或开仓动作之前。
  - 为开仓补齐无副作用的 entry swap 预览能力。
  - 让稳定币到稳定币的 entry swap 默认使用独立推荐滑点，而不是直接继承全局滑点。
  - 要求用户在看到预计到账数量和本次滑点后，明确确认，才允许执行真实兑换。
  - 保证真实执行时顺序固定为“先 entry swap，后开仓”。
- Non-Goals:
  - 不在本次变更中重做 Zap 内部二次配比 swap 逻辑；本次只覆盖开仓前的钱包侧 entry swap。
  - 不在本次变更中引入新的全局滑点配置项；本次先按后端推荐策略返回推荐值。
  - 不在本次变更中重做任务再平衡的 UI；本次只覆盖手动开仓入口。

## Decisions
- Decision: 风险校验优先级最高，必须先于 entry swap 预览和真实执行
  - 统一风险校验至少包括：
    - 池子价格与 OKX 价格偏差检查
    - 链上 raw liquidity 检查
    - 美元流动性及其相关限额/确认规则
    - 现有开仓链路里的其他结构化安全拦截
  - `open_position_preview` 必须先跑这套风险校验；任何 blocker 命中时，直接返回结构化风险结果，不再继续 entry swap 预估。
  - `open_position` 真正执行前也必须再次跑同一套风险校验，避免预览后到执行前市场状态变化导致漏拦截。
  - 风险校验未通过时：
    - 不得继续 entry swap 预览确认流程
    - 不得执行真实 entry swap
    - 不得创建持久化任务

- Decision: 新增 `open_position_preview`，专门负责 entry swap 预览
  - 预览接口不创建任务、不发链上交易、不部署私有合约。
  - 预览接口在进入 entry swap 预估前，必须先完成统一风险校验。
  - 预览接口复用当前开仓表单参数和钱包选择，返回：
    - 风险校验是否通过
    - 是否需要前置 entry swap
    - entry swap 的 `from token` / `to token`
    - 预计输入数量、预计到账数量
    - 推荐滑点
    - 当前用于预览的生效滑点
  - 若当前钱包已有足够 entry token，预览接口返回“不需要前置兑换”，页面无需增加额外确认步骤。

- Decision: 执行接口在需要前置兑换时强制要求显式确认
  - `open_position` 新增显式确认字段，例如 `confirm_entry_swap`。
  - `open_position` 在任何真实执行前，必须再次先完成统一风险校验。
  - 当后端判断本次请求需要前置 entry swap 且未确认时：
    - 不得执行真实兑换
    - 不得继续后续开仓
    - 不得创建持久化任务
    - 必须返回结构化的“需要确认前置兑换”响应，供客户端展示
  - 这样即使旧客户端未接入预览接口，也不会再直接用默认滑点执行真实 entry swap。

- Decision: 稳定币到稳定币的推荐滑点独立于全局滑点
  - 对 `USDT -> USDC`、`USDT -> BUSD` 等稳定币到稳定币的 entry swap，后端必须给出独立推荐滑点。
  - 该推荐值不得简单等于用户全局滑点，尤其在全局滑点更高时。
  - 用户可以覆盖本次滑点；客户端修改后必须重新请求预览，确保页面展示的预计到账数量与当前滑点一致。

- Decision: 真实执行严格遵循“先 swap，后开仓”
  - 用户确认后，后端先执行 entry swap。
  - 只有 entry swap 成功后，才继续后续私有 Zap / 开仓。
  - 如果 entry swap 失败，后端必须停止流程，不得继续开仓。

## Risks / Trade-offs
- 增加了预览阶段，请求次数会变多。
  - Mitigation: 只在表单参数完整且需要前置兑换时触发预览；客户端可显式触发或在关键字段变更后刷新。
- 预览报价和真实执行报价之间仍可能有时间差。
  - Mitigation: 预览只作为用户确认依据；执行前后端仍按用户确认的滑点重新获取实时 OKX 数据。
- 如果 entry swap 成功但后续开仓失败，钱包里会留下 entry token。
  - Mitigation: 这是当前两段式链路的既有特性；本次只要求页面和错误提示明确反映阶段状态，不额外引入自动回滚。

## Migration Plan
1. 先统一抽出开仓风险校验，并接入预览和执行两条链路，确保所有风险都在最前面被拦截。
2. 再落后端预览模型与执行确认门槛，确保未确认时不会触发真实兑换。
3. 再接 MiniApp / WebApp 的预览展示、滑点修改和确认交互。
4. 最后回归验证“风险直接拦截”、“无需前置兑换”和“需要前置兑换”三条路径。

## Open Questions
- 暂不在提案里写死稳定币到稳定币的推荐滑点具体数值；实现时由后端策略统一返回，但必须满足“不可直接沿用全局高滑点”的约束。
