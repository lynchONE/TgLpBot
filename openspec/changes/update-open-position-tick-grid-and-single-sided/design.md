## Context
当前开仓后端已经支持按百分比、Tick、格子解析区间，并且会在预览结果中返回规范化后的 Tick、价格映射和 `single_token0` / `single_token1` / `dual_sided` 仓位形态。现在缺的不是新的执行协议，而是一层更贴近用户心智的前端编辑体验。

用户在单边仓场景下更倾向于直接输入价格区间，例如“当前价 0.1，我要做 0.05 - 0.07”，以及一键把当前区间整体挪到当前价下方或上方。只提供 Tick 输入虽然可行，但心智成本偏高。

## Goals / Non-Goals
- Goals:
  - 在不新增后端 `price` 模式的前提下，支持按价格直接编辑开仓区间
  - 明确展示当前将开单边仓还是双边仓，并说明主偏向资产
  - 提供 `单边下限` / `单边上限` 快捷操作，降低单边区间调整成本
  - 保持 WebApp 与 MiniApp 的区间语义一致
- Non-Goals:
  - 不新增后端 `price_lower` / `price_upper` 请求字段
  - 不改动现有实际执行链路，最终仍以规范化后的 Tick 区间为准
  - 不引入新的图表组件或完整 K 线编辑器

## Decisions
- Decision: 价格输入是前端编辑模式，不是新的后端协议
  - WebApp 与 MiniApp 新增本地 `price` 编辑模式。
  - 前端使用当前计价方向、token decimals 与 `tick = log(price / 10^(decimalsAdj)) / log(1.0001)` 将价格换算为 Tick。
  - 预览与提交仍然调用现有 `tick_lower` / `tick_upper` 流程，由后端做最终规范化。
  - Alternatives considered: 为后端新增 `price` 输入模式。
  - Rationale: 后端已有 Tick 规范化与预览回显能力，前端本地换算更小、更稳，也更符合这次需求的边界。

- Decision: 单边快捷按钮通过“保留宽度，整体平移”实现
  - `单边下限`：保留当前宽度，将区间整体移动到当前价锚点下方。
  - `单边上限`：保留当前宽度，将区间整体移动到当前价锚点上方。
  - 若当前宽度无效，则回退为 1 个 `tick_spacing` 宽度。
  - Alternatives considered: 只做文案提示，不提供快捷按钮。
  - Rationale: 用户的目标是快速把区间放到当前价外侧，保留宽度比重置成固定模板更符合预期。

- Decision: 单边/双边结果提示升级为显式摘要层
  - 在区间编辑区附近显示当前仓位形态状态卡。
  - 单边时明确展示“从 USDT 入场，最终将主要持有某一侧资产”。
  - 双边时展示当前将形成双边仓，并保留价格区间和 Tick 区间摘要。
  - Alternatives considered: 仅在底部确认区保留文字说明。
  - Rationale: 单边/双边是用户操作区间时最关心的信息，应该尽早回显，而不是等到最后确认才看见。

## Risks / Trade-offs
- 风险: 价格输入在不同计价方向下容易产生“上下限显示方向”和 Tick 方向不一致的理解偏差
  - Mitigation: 价格输入与图表使用同一计价方向，并始终以预览返回的规范化价格区间做最终回显
- 风险: 极小价格或极大价格换算 Tick 时可能出现精度误差
  - Mitigation: 前端只负责近似映射，最终以预览返回的规范化 Tick 和价格区间作为真值
- 风险: 快捷按钮如果直接重置为固定模板，容易让用户丢失已调好的宽度
  - Mitigation: 快捷按钮默认保留当前宽度，仅做位置平移

## Migration Plan
1. 先补充前端价格编辑与单边快捷层，继续复用已有预览/开仓接口。
2. 预览返回的 `range_editor` 继续作为规范化结果源，价格输入只用于本地编辑。
3. WebApp 与 MiniApp 同步落地后，再评估是否需要把相同能力迁移到任务区间编辑。
