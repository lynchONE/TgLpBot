## Context
- 当前 `sm_wallet_daily_snapshots` 既承载每日聚合结果，又被实时排行榜当作“今日 0 点基准”读取。
- `tryAggregatePreviousDay` 的完成标记只存在于进程内存，服务在 00:05 之后重启会再次执行昨日聚合。
- `captureSmartMoneyWalletSnapshots` 当前会删除指定 `snapshot_day` 的旧行，并用执行时刻的实时资产重新写入，因此会把原始基准覆盖成日内最新值。

## Goals / Non-Goals
- Goals:
  - 为聪明钱资产提供独立、不可被日内聚合覆盖的 0 点资产快照。
  - 支持至少 31 天的 0 点快照查询和保留。
  - 实时排行榜使用“最新实时资产 - 当日 0 点快照”。
  - 盈亏日历使用“当日 0 点快照 - 前一日 0 点快照”。
- Non-Goals:
  - 不重做链上资产识别口径。
  - 不改变 `sm_lp_daily_stats` 的 LP 事件统计职责。
  - 不回填已经被覆盖且无历史来源可恢复的旧 0 点快照。

## Decisions
- Decision: 新增独立 0 点快照表
  - 使用新表而不是继续复用 `sm_wallet_daily_snapshots`，避免历史聚合、排行榜缓存刷新和服务重启重跑互相影响。
  - 新表按 `wallet_address + chain_id + snapshot_day` 建唯一索引。

- Decision: 0 点快照写入必须幂等
  - 普通定时任务发现当日 0 点快照已存在时直接跳过，不覆盖旧值。
  - 如后续需要人工修复历史数据，应使用显式 backfill / force 工具，而不是复用普通调度路径。

- Decision: 日历资产变化优先使用相邻 0 点快照
  - 当日盈亏日历中的资产变化值来源为当日 0 点总资产减前一日 0 点总资产。
  - 若缺少前一日或当日 0 点快照，则该日资产变化按 0 或缺失态降级，不使用日内实时资产补算。

- Decision: 实时榜基准使用当日 0 点快照
  - 排行榜 `pnl` / `yield_rate` 的实时口径使用最新 `sm_wallet_live_states` 与当日 0 点快照计算。
  - 响应继续返回 `baseline_total_usd` 和 `current_total_usd`，用于前端显示“0点”和“最新”金额。

## Risks / Trade-offs
- 风险: 如果服务在 0 点附近不可用，当日 0 点快照可能缺失。
  - Mitigation: 启动后若发现当日 0 点快照缺失，可以尽快补写一次，并通过 `captured_at` 暴露实际采集时间；但不得覆盖已存在快照。
- 风险: 历史上已被覆盖的旧数据无法从当前表自动恢复。
  - Mitigation: 对已覆盖日期只能从备份、外部历史缓存或重新等待后续自然日产生正确快照。
- 风险: 0 点资产快照和 LP 事件统计的自然日边界需要一致。
  - Mitigation: 统一使用 `timeutil.Location()` 与 `dayStart` / `formatDay`。

## Migration Plan
1. 新增 0 点快照模型并加入自动迁移。
2. 新增 0 点快照捕获任务，普通调度路径只插入缺失快照，不覆盖已有快照。
3. 调整实时排行榜、总览钱包基准、钱包详情基准读取新表。
4. 调整聪明钱历史和钱包盈亏日历查询，按相邻 0 点快照计算资产变化。
5. 增加 31 天保留清理逻辑与相关测试。
