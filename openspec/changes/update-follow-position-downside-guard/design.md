## Context
跟单任务创建时会写入 `is_follow=true`，并默认使用 `exit_all` 作为任务模式；但策略轮询中目前对 `is_follow` 任务直接跳过出区间处理，实际平仓主要依赖自动跟单 worker 捕获目标钱包撤仓事件。

用户需要在保留跟单语义的同时补一条资金安全底线：目标钱包撤仓事件漏掉时，价格若向下跌破跟单仓位区间，系统应直接撤出，避免持续吃跌。

## Goals / Non-Goals
- Goals: 跟单仓位下破区间时立即撤出；WebApp 和 MiniApp 清楚展示跟单仓位的真实自动处理规则；测试覆盖上破不处理、下破撤出。
- Non-Goals: 不新增用户可配置的下破阈值；不改变普通仓位三种区间策略；不让跟单仓位上破自动再平衡或自动撤出。

## Decisions
- Decision: 跟单仓位下破保底使用策略轮询处理，而不是依赖跟单事件 worker。
  - 这样可以复用现有 tick 获取、任务锁、撤仓到 USDT 和停止任务流程。
  - 只在 `is_follow=true` 且价格方向为下破时触发，避免影响普通任务。
- Decision: 下破保底立即执行，不等待 `reopen_delay_seconds`。
  - 跟单仓位的主要退出信号是目标钱包撤仓事件；下破保底是风险兜底，应尽快减少继续下跌暴露。
- Decision: 前端展示跟单专用策略说明，不复用普通任务的“双向撤出/双向再平衡/上破再平衡”按钮语义。
  - MiniApp 和 WebApp 仓位卡片都需要让用户看到：目标撤仓会跟随，下破会保底撤出，上破不会自动处理。

## Risks / Trade-offs
- 风险：短暂下破后又快速回区间，立即撤出可能错过后续回升。
  - Mitigation: 这是保底策略，优先控制下行风险；上破仍不自动处理，保持跟单语义。
- 风险：撤仓执行失败可能留下风险敞口。
  - Mitigation: 复用现有撤仓重试/错误状态路径，并在任务状态和通知里保留失败原因。

## Migration Plan
1. 不需要新增数据库字段；现有 `is_follow`、tick 区间和任务状态足够支撑。
2. 更新策略轮询逻辑和单元测试。
3. 更新实时仓位 DTO 或前端推导字段，用于 WebApp/MiniApp 展示跟单策略说明。
4. 更新前端仓位卡片。
