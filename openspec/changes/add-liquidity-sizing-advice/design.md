# Design: 加仓金额建议与边际效率控制

## Context
- 用户需要在真正执行开仓前，先知道“为了拿到某个区间占比，需要投入多少钱”，并判断这笔资金是否值得。
- 用户不希望额外手填价格、tick、资金规模和风控参数；开仓交互应继续保持“金额 + 区间 + 滑点”的现有模式。
- 当前项目已有开仓前流动性校验和基础风控能力，但没有独立的“仓位规模建议”计算层。
- 这类建议如果被钱包余额或比例风控直接截断，很容易把三档结果压成同一个数，前端展示上几乎不可用。

## Goals / Non-Goals
- Goals:
  - 在不改变现有开仓请求形态的前提下，自动补齐建议计算所需输入
  - 基于统一公式输出可解释的保守 / 中性 / 激进三档仓位规模建议
  - 对高占比目标自动做边际效应抑制，避免 80% 以上资金效率明显下降
  - 保证建议结果不依赖当前钱包余额，保留三档建议的区分度
  - 输出结构稳定的 JSON，便于 `open_position_preview`、MiniApp、WebApp、Bot 复用
- Non-Goals:
  - 不在 V1 中估算链上精确的 token0 / token1 构成
  - 不在 V1 中直接改写用户提交的开仓金额或静默替用户执行交易
  - 不在 V1 中处理多区间、多仓位组合优化

## Decisions
- Decision: 复用现有开仓预检链路输出建议，而不是新增一套要求用户显式传参的独立接口
  - 原因：用户已经明确希望开仓方式不变，建议能力应成为开仓预览的一部分。

- Decision: 建议计算所需输入由后端自动解析
  - `price` / `current_tick`：沿用现有开仓预览里的链上读取逻辑
  - `tick_lower` / `tick_upper`：沿用现有区间百分比转 ticks 的计算过程
  - `L_active`：优先 `pools.active_liquidity_usd`，其次其他池子流动性估值来源
  - `target_share_min/max`：优先用户全局配置，缺省回退系统配置

- Decision: 三档目标占比分别使用固定基准再落入配置区间
  - `conservative` 基准目标为 `0.20`
  - `neutral` 基准目标为 `0.40`
  - `aggressive` 基准目标为 `0.65`
  - 每档都会先与配置中的 `[target_share_min, target_share_max]` 做夹取，再应用 `0.8` 的全局上限
  - 如果夹取后多档结果重叠，仍保留三档输出，但在说明里标明“目标区间过窄导致档位收敛”

- Decision: 推荐金额只由活跃流动性和目标占比反推，不再被钱包余额或比例风控截断
  - 公式：`user_liquidity = L_active * target_share / (1 - target_share)`
  - 这样返回的是“达到目标占比所需规模建议”，而不是“当前钱包一定能立刻执行的金额”
  - 真实下单阶段仍沿用既有余额、滑点、价格偏差等校验逻辑，不在建议层重复裁剪

- Decision: 边际效率评价按目标占比区间判断，实际占比单独重算
  - `high`：目标占比 `<= 0.5`
  - `medium`：目标占比 `> 0.5` 且 `<= 0.8`
  - `low`：请求目标占比 `> 0.8`，进入低效率区间并被系统截断

- Decision: `risk_exposure` 采用保守近似
  - 在缺少 token 细分和后续价格路径输入时，V1 将“最大可能单边资产”定义为“本次建议投入金额在极端情况下可能全部集中到单边资产”，因此 `risk_exposure ~= liquidity_to_add`
  - 该值用于做上限提示，不宣称为链上精确估值

## Risks / Trade-offs
- 如果调用方把建议结果误当成“直接可执行金额”，会出现建议金额高于钱包余额的情况，因此 UI 必须保持“建议”语义，而不是“自动代填”语义。
- `aggressive` 档默认使用 `0.65` 作为中心值，简化了 60% 到 70% 的多档展开；这样接口更稳定，但表达粒度略粗。
- 采用“单边资产≈投入金额”的风险暴露近似会偏保守，但比给出看似精确、实际依赖未提供输入的数字更稳妥。

## Migration Plan
1. 新增纯计算逻辑与自动输入解析。
2. 在现有开仓预览响应中增加建议结构化 JSON。
3. 补充全局 / 系统默认值读取逻辑。
4. 为目标占比截断、档位收敛、无钱包余额依赖和自动数据回退补充单元测试。

## Open Questions
- V1 暂不扩展“当前已持有该区间仓位金额”输入，默认把本次建议金额视为从零开始达到目标占比所需的规模建议；如果后续需要“增量补仓建议”，应另起变更补充已有仓位输入。
