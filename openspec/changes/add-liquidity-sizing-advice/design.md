# Design: 加仓金额建议与边际效率控制

## Context
- 用户需要在真正执行开仓前，先知道“为了拿到某个区间占比，需要投入多少钱”，并判断这笔资金是否值得。
- 用户不希望额外手填价格、tick、资金规模和风控参数；开仓交互应继续保持“金额 + 区间 + 滑点”的现有模式。
- 当前项目已有开仓前流动性校验和风险上限等能力，但没有独立的“仓位规模建议”计算层。
- 输入里已经直接给出 `L_active`（U 计价的当前区间活跃流动性），因此 V1 不需要再从 `price` 和 tick 范围去推导池子资金规模；它们更多是上下文信息和未来扩展位。

## Goals / Non-Goals
- Goals:
  - 在不改变现有开仓请求形态的前提下，自动补齐建议计算所需输入
  - 基于统一公式输出可解释的保守 / 中性 / 激进三档仓位规模建议
  - 对高占比目标自动做边际效应抑制，避免 80% 以上资金效率明显下降
  - 同时遵守单仓风险上限和用户总资金上限
  - 输出结构稳定的 JSON，便于 `open_position_preview`、MiniApp、WebApp、Bot 复用
- Non-Goals:
  - 不在 V1 中估算链上精确 token0 / token1 组成
  - 不在 V1 中直接改变用户提交的开仓金额或静默替用户执行交易
  - 不在 V1 中处理多区间、多仓位组合优化

## Decisions
- Decision: 复用现有开仓预检链路输出建议，而不是新增一套用户显式传参的独立接口
  - 原因：用户已经明确希望开仓方式不变，只填写金额、区间、滑点；建议能力应成为开仓预览的一部分。

- Decision: 建议计算所需输入由后台自动解析
  - `price` / `current_tick`：沿用现有开仓预览里从链上读取 current tick / slot0 的逻辑
  - `tick_lower` / `tick_upper`：沿用现有 `range_lower_pct` / `range_upper_pct` 到 ticks 的计算过程
  - `L_active`：优先 `pools.active_liquidity_usd`，其次 `current_pool_value` / `reserve_in_usd`，最后 DexScreener
  - `capital_total`：优先使用选中钱包在当前链上的 stable 资产估值
  - `target_share_min/max` 与风险参数：优先用户全局配置，缺省回退系统配置

- Decision: 三档目标占比分别使用固定基准再落入用户目标范围
  - `conservative` 基准目标为 `0.20`
  - `neutral` 基准目标为 `0.40`
  - `aggressive` 基准目标为 `0.65`，代表 60%–70% 区间的中位建议
  - 每档都会先与配置中的 `[target_share_min, target_share_max]` 做夹逼，再应用 `0.8` 的全局上限
  - 如果夹逼后多档结果重叠，仍保留三档输出，但在说明里标明“目标区间过窄导致档位收敛”

- Decision: 风险上限拆成“固定金额上限 + 比例上限”两类配置，再合成为有效上限
  - 固定金额上限：例如 `500U`
  - 比例上限：例如 `20% * capital_total`
  - 实际有效上限取“理论金额、固定金额上限、比例金额上限、capital_total”中的最小值
  - 这样可以覆盖“500U 或总资金 20%”这一类口径，而不需要用户每次输入

- Decision: 边际效率评价按目标区间判断，实际占比单独重算
  - `high`：目标占比 `<= 0.5`
  - `medium`：目标占比 `> 0.5` 且 `<= 0.8`
  - `low`：用户请求目标占比 `> 0.8`，进入低效率区间并被系统截断
  - 若建议金额被资金/风险上限进一步截断，则 `expected_share` 按截断后的实际建议金额重新计算

- Decision: `risk_exposure` 采用保守近似
  - 在缺少 token 细分和后续价格路径输入时，V1 将“最大可能单边资产”定义为“本次建议投入金额在极端情况下可能全部集中到单边资产”，因此 `risk_exposure ~= liquidity_to_add`
  - 该值用来做上限提示，不宣称为链上精确估值

- Decision: 返回结构中增加解释字段
  - 除 `recommended_positions` 和 `warnings` 外，响应还应返回规范化后的关键输入和每档计算摘要，确保调用方能解释推荐值来源
  - 当建议计算数据缺失时，响应应返回 `warnings` 并允许建议段整体降级为空，而不破坏现有开仓预检主响应

## Risks / Trade-offs
- 如果调用方传入的 `L_active` 不准确，建议金额也会随之偏移；V1 假设该输入已经是可信的区间活跃流动性估值。
- `aggressive` 档默认使用 `0.65` 作为中心值，简化了 60% 与 70% 两个独立建议的输出；这样接口更稳定，但表达粒度略粗。
- 采用“单边资产≈投入金额”的风险暴露近似会偏保守，但比给出看似精确、实际依赖未提供输入的数字更稳妥。
- 如果把 `capital_total` 定义成全用户总资产，会高估当前钱包当下可执行的预算；因此 V1 有意收窄到“当前选中钱包的 stable 资金规模”。

## Migration Plan
1. 新增纯计算逻辑与自动输入解析。
2. 在现有开仓预览响应中增加建议结构化 JSON。
3. 补充全局/系统默认值读取逻辑。
4. 为目标占比截断、风险上限截断、总资金截断、自动数据回退和档位收敛补充单元测试。

## Open Questions
- V1 暂不扩展“当前已持有该区间仓位金额”输入，默认把本次建议金额视为本次计划投入金额对应的建议上限；如果后续需要“增量补仓建议”，应另起变更补充已有仓位输入。
