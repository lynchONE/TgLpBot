## Context
- 当前钱包一键兑换执行接口在后端已经等待交易回执，但前端仍然把成功结果当成“已提交”，缺少明确完成语义。
- 项目里已经有通用 `transactions` 表，并且其他链上流程会记录交易历史，可以复用为钱包兑换历史的数据源。
- 本次需求同时涉及后端响应模型、交易落库、历史查询和前端状态管理，属于跨模块改动。

## Goals / Non-Goals
- Goals:
  - 让钱包一键兑换成功后展示明确的完成反馈，而不是模糊的提交态
  - 让前端在成功后清理执行态并刷新相关数据，避免下一笔操作残留旧状态
  - 为当前钱包提供最近兑换历史展示
- Non-Goals:
  - 不实现全局分页历史中心
  - 不新增独立的兑换历史表
  - 不改变 OKX 路由或链上执行语义

## Decisions
- Decision: 钱包一键兑换历史复用现有 `transactions` 表
  - 使用 `type=swap`、`user_id`、`chain`、`from_address` 过滤当前钱包兑换记录
  - 通过 `tx_hash` 去重，避免重复写入同一笔兑换
- Decision: 执行接口返回值以“已完成”为准
  - 因为后端在返回前已经等待交易回执，所以前端成功态必须使用完成语义
  - 返回结构中补充 `tx_url`、`completed_at`、`to_amount`、`to_amount_float`
- Decision: 前端成功后立即刷新钱包余额和最近兑换
  - 兑换成功后清空旧报价和执行中状态
  - 复用已有余额刷新逻辑，同时拉取最近兑换历史

## Risks / Trade-offs
- 交易历史依赖执行接口成功后落库，若落库失败，用户仍可完成兑换但历史列表暂时缺失该笔记录
  - Mitigation: 落库失败不影响主流程，但保留日志并允许用户通过刷新再次检查后续修复结果
- 历史列表展示需要补 token metadata，部分代币可能暂时只能展示地址或基础 symbol
  - Mitigation: 前端和后端都提供 symbol/name/logo 的 best-effort 回退

## Migration Plan
1. 扩展钱包单币兑换执行返回模型，并在成功后记录交易。
2. 新增当前钱包兑换历史接口，并挂到 `settings` 兼容路由和独立 API 路由。
3. 前端接入成功卡片、状态清理和历史列表。
4. 回归验证后端测试和 `webapp` 构建。
